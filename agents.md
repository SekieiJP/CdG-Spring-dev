# プロジェクト: カードで学習塾 (Webブラウザ版)

## 1. 目的
Webブラウザ上で動作するスマートフォン向けのカードゲーム「カードで学習塾 春期講習編」を制作すること。
本プロジェクトは、以下のドキュメントに基づき、アナログボードゲームのルールをデジタルゲームとして再現することを目的とします。

## 2. 参照資料
*   **ルールブック:** @[カードで学習塾春期講習編ルールブック260204a.md]
*   **カードリスト:** @[cards_fresh.csv] @[cards_pro.csv]

## 3. ゲーム概要
プレイヤーは学習塾の室長となり、春期シーズン（全8ターン）を通じて教室運営を行います。
ステータス（体験・入塾・満足・経理）を管理し、アクションカードを駆使して目標ポイントの最大化を目指します。

## 4. 具体的要件

以下の要件を開発プロセス全体に反映します。

### 4.1 開発プロセス要件（品質管理）
*   **テスト駆動開発 (TDD):**
    *   コアロジック（ターン進行、カード効果、スコア計算等）には単体テストを作成。
    *   リファクタリングやカード追加時のデグレーションを防止。
*   **段階的プロトタイピング:**
    *   未実装カードや効果があっても、最初から最後まで一貫してゲームが動作する構造。
    *   カード効果は汎用パーサーで処理し、未実装効果は警告を出しつつスキップ可能に。
*   **詳細ログ出力:**
    *   ゲーム進行中、全アクション・ステータス変化を画面内にログ表示。
    *   不具合発生時の原因追跡を容易にする。

### 4.2 デプロイメント要件
*   **シンプルなデプロイ構成:**
    *   1フォルダに全ファイル（HTML, CSS, JS, CSV）を配置。
    *   Webサーバにアップロードするだけで動作（外部依存なし、CDNのみ許可）。

### 4.3 UI/UX要件
*   **スマートフォン縦画面対応:**
    *   レスポンシブデザインで、縦画面を主軸に最適化。
*   **操作方法の柔軟性:**
    *   **ドラッグ＆ドロップ:** カードを直接スタッフ枠にドラッグ。
    *   **タップ順操作:** カードをタップすると自動的に空き枠（室長→講師→事務の優先順）へ配置。
*   **カテゴリ別カラースキーム（色覚多様性配慮）:**
    *   **動員:** 青系（例: `#3B82F6` Blue-500）
    *   **教務:** 緑系（例: `#10B981` Green-500）
    *   **庶務:** 桃色系（例: `#EC4899` Pink-500）
    *   **応対:** オレンジ系（例: `#F97316` Orange-500）
    *   色覚多様性のため、アイコンやパターンでも区別。

### 4.4 機能要件
*   **スコア記録・比較機能:**
    *   **ローカルストレージ:** 自身のハイスコア記録（過去プレイの最高記録を保持）。
    *   リプレイ性を高め、スコアアタックを促進。

### 4.5 研修カード抽選仕様
*   **研修候補プール:**
    *   データリストに掲載されたカードを各レアリティごとに各2枚ずつ搭載した内部プールをもつ。
*   **抽選ルール:**
    *   毎回の研修では、プールのうち指定されたレアリティのカードから、異なる3種類のカードを提示する。
    *   同名カードが2枚とも提示されることはない。
*   **プールからの除外:**
    *   提示された3枚のカードは研修候補プールから除外される。
    *   その結果、次以降の研修で同じカードが出現する確率が減少する。
*   **リセット:**
    *   研修候補プールは、新しいゲームを開始するたびリセットされる。

## 6-A. 難易度システム設計指針

### 概要
本ゲームは「FRESH」（基本）と「PRO」（上級）の2つの難易度を持つ。カードプールは難易度ごとに完全に別管理（別CSVファイル）とし、同名カードでも難易度によって効果テキストが異なる場合がある。パース規則は共通エンジンを使用する。

### 難易度選択
- タイトル画面で選択。PRO難易度は初回から選択可能
- 初訪問者(`!localStorage.getItem('cdg_visited')`)にはFRESHに「初めての方はこちらに挑戦」吹き出しを表示

### JSファイル機能一覧と難易度対応の影響

| ファイル | 責務 | 難易度対応の影響 |
|---|---|---|
| `main.js` | エントリーポイント。カードCSV読込パス指定、ゲーム初期化、セーブ復元。デバッグ関数提供 | 難易度に応じたCSVパス切替、難易度選択フローの起点 |
| `gameState.js` | プレイヤーステータス(体験/入塾/満足/経理)管理、デッキ/手札/配置操作、境界値チェック | 難易度フィールド追加、初期ステータスの難易度分岐、トークン管理、リフレッシュ回数管理、`placed`を配列構造に変更 |
| `cardManager.js` | CSVパース、カード効果テキストのパース(【】スタッフ制限/〈〉条件付き効果)、研修候補プール管理 | 難易度別CSV読込対応、新効果キーワード(情熱/発想/整理/並行/研鑽/疲労)のパーサー拡張、レアリティ「T」とカテゴリ「支障」追加 |
| `turnManager.js` | 8ターンの設定(研修レアリティ/おすすめ行動/削除枚数)、フェーズ遷移、アクション実行 | ターン設定の難易度別定義、トークン消費ロジック(ドロー枚数/候補枚数/削除上限の動的変更) |
| `scoreManager.js` | スコア計算(退塾/動員/入退目標)、ランク判定(E〜S+)、ハイスコアのLocalStorage管理 | スコア計算式の難易度別分岐(FRESH: S+満点10、小数1桁四捨五入)、ハイスコアの難易度別キー管理 |
| `uiController.js` | 全画面UI制御(68メソッド): 研修/行動/会議/結果/設定画面。ドラッグ&ドロップ/タップ操作 | 難易度選択UI(初訪問者吹き出し)、結果画面の得点表示分岐、研修リフレッシュUI、ステータス目標表示(両難易度) |
| `saveManager.js` | ゲーム状態のLocalStorage永続化・復元(中断再開機能)。ビルドバージョン管理 | セーブデータに難易度・トークン・リフレッシュ回数追加、難易度別の復元ロジック |
| `logger.js` | ゲームログの記録・表示 | 変更不要(新効果のログは各マネージャーが出力) |

### 新効果システム（トークン方式）
新効果はトークン（チップ）を獲得する形で表現。該当フェーズが到来した時に発動し、発動後にトークンは破棄される。効果は累積する。

| 効果 | 発動タイミング | 内容 |
|---|---|---|
| ✨情熱 | 次のターンのドロー時 | 手札ドロー+N |
| 💡発想 | 次のターンの研修 | R,SR,SSR各1枚から1枚を追加習得×N回 |
| 🚥整理 | 同ターンの会議 | 削除上限+N |
| 🤹並行 | 即時（配置時判定） | 配置済みスタッフにも配置可 |
| 💤疲労 | 次のターンのドロー時 | 手札ドロー-N |

### ステータス目標表示（両難易度共通）
- **体験/入塾**: 項目別ランク + プログレスバー + 「あとN」の3要素を並列表示
- **経理/満足**: プログレスバー + 「あとN」の2要素を並列表示

### 設計原則

1. **難易度設定の一元管理**: 難易度ごとの差異（初期値、CSV、スコア閾値、ターン設定など）は `difficultyConfig.js`（新規）に集約し、各マネージャーはconfigを参照する
2. **カードプールの分離**: FRESHとPROで別CSVファイルを使用(`data/cards_fresh.csv`, `data/cards_pro.csv`)。共通カードも各ファイルに個別記載
3. **パーサーの共通化**: `cardManager.parseEffect()` は難易度問わず同じテキスト→同じ効果を返す。新効果キーワードはパーサーに追加するだけで両難易度で利用可能
4. **ハイスコアの難易度別管理**: LocalStorageキーに難易度サフィックスを付与（例: `bdrinkai_highscore_fresh`）
5. **セーブデータの後方互換**: 旧データ（難易度フィールドなし）は「FRESH」として復元

### 詳細仕様
確定済み仕様と後日提供データの一覧は `/readyToCreatePRODifficulty.md` を参照。

## 6. キャッシュバスター管理

ブラウザキャッシュを確実に無効化するため、ファイルを更新した際は以下の**全箇所**のキャッシュバスターを同じ値に揃えること。

### 記載場所一覧

| ファイル | 行 | 内容 |
|----------|----|------|
| `game/index.html` | L9 | `<link rel="stylesheet" href="css/style.css?v=XXXX">` |
| `game/index.html` | L233〜234 | コメントと `<script src="js/main.js?v=XXXX">` |
| `game/js/main.js` | L5〜12 | 各モジュールのimport文 `?v=XXXX` (8行、difficultyConfig.js含む) |
| `game/js/main.js` | L14 | `const CACHE_BUSTER = 'vXXXX'` |

> **Note**: `releaseNote.html` 内のバージョン番号はリリースノートの履歴（変えない）。キャッシュバスターの対象外。

### 更新手順

1. 現在日時を `YYYYMMDD-HHMM` 形式で決定する（例: `20260219-0141`）
2. 以下のgrepコマンドで現在値を確認する:
   ```
   grep -rn "?v=\|CACHE_BUSTER" game/index.html game/js/main.js
   ```
3. 上記「記載場所一覧」の全箇所を同じ値に更新する

### 検索コマンド（更新漏れ確認用）

```bash
# キャッシュバスターの全記載を検索（node_modulesを除く）
grep -rn "?v=20" game/ --include="*.html" --include="*.js" --exclude-dir=node_modules

# 値が統一されているか確認（異なる値があれば複数行表示される）
grep -rn "?v=20\|CACHE_BUSTER" game/index.html game/js/main.js
```
